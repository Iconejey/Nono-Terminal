# PLAN.md: Nono-Terminal Architectural Design & Implementation Plan

This document outlines the research, system architecture, and step-by-step plan for building **Nono-Terminal** using **OpenAI** (via the official `openai` npm package) for the AI assistant agent, rather than Google Gemini.

---

## Architecture & Technical Stack

Nono-Terminal is built on Electron. It isolates the browser UI from the local system using a preload script that exposes safe IPC APIs to the renderer.

```
┌────────────────────────────────────────────────────────┐
│                   Renderer (UI Layer)                  │
│             index.html | style.css | window.js         │
└───────────────────────────┬────────────────────────────┘
                            │ (Safe IPC via preload.js)
┌───────────────────────────▼────────────────────────────┐
│              Main (Multi-Window Controller)            │
│  - App Lock | Window Registry Map | OpenAI API Client  │
└───────────────┬──────────────────────────┬─────────────┘
                │                          │
┌───────────────▼──────────────┐   ┌───────▼──────────────────────┐
│    Window 1 Session          │   │    Window 2 Session          │
│    - Shell: /bin/bash (CWD1) │   │    - Shell: /bin/bash (CWD2) │
│    - OpenAI Conversation 1   │   │    - OpenAI Conversation 2   │
└──────────────────────────────┘   └──────────────────────────────┘
```

### Stack Components:

1. **Frontend:** Vanilla HTML, CSS (Consolas/Material Symbols Outlined font), and JavaScript.
2. **IPC Preload:** A script `preload.js` that uses `contextBridge` to expose functions for sending shell inputs and receiving terminal stdout streams.
3. **Backend (Main process):** Manages a single-instance app lock. Keeps a registry map of active windows and maps incoming IPC events to window-specific shell and OpenAI conversation instances using `event.sender.id`.
4. **OpenAI SDK:** The official `openai` package to handle tool calling and streaming.

## Core Features & Research Results

### 1. Persistent Terminal Execution

To keep terminal state (like environment variables, active processes, and current working directories) intact between command invocations:

- **Strategy:** We spawn a persistent shell process (e.g. `bash` or `/bin/sh`) in `main.js`.
- **Completion & CWD tracking:** After every command is typed by the user (or generated by the agent), we append a unique tracking suffix:
    ```bash
    ; echo "__NONO_CMD_END__ $? $PWD"
    ```
- **Parsing:** We pipe `stdout` and `stderr` through a line-buffer. When the delimiter is found, we extract the exit status (`$?`) and the updated directory (`$PWD`), notify the UI via IPC, and hide the delimiter line from the user.

### 2. Command Interruptions & Timeout Controls

- **Ctrl+C Handling:** To support interrupting running commands (like `npm install` or hanging loops) using `Ctrl+C`, the renderer will send a signal to the main process, which writes `\x03` (the ETX/Ctrl+C control character) to the shell's `stdin`. This sends a SIGINT to the active foreground command without killing the parent shell.
- **Agent Command Timeouts:** Commands run by the AI agent will have a maximum timeout (e.g., 30 seconds) to prevent the terminal from freezing on infinite scripts or interactive prompt blocks.
- **OpenAI Request Timeout & Retry:** To prevent locking or freezing on hanging API calls, we will implement a constant timeout (e.g., `const API_TIMEOUT_SECONDS = 30;`). If an OpenAI request takes longer than this timeout, we abort the request using `AbortController` and seamlessly retry the call (up to 3 attempts) before reporting a connection issue to the user.

### 3. OpenAI Agent Tool Integration

We will configure the OpenAI API with function/tool definitions.

#### Available Tools:

- `execute_command(command)`: Runs a command in the persistent shell and returns stdout/stderr.
- `read_file(path, start_line, end_line)`: Reads lines from a file.
- `edit_file(path, search_content, replace_content)`: Performs flexible search-and-replace blocks inside a file rather than full-file rewrites (avoiding LLM whitespace/indentation mismatches).
- `search_codebase(query)`: Native grep-like tool that searches files for matches and returns a list of matching lines and paths (so the LLM doesn't have to construct fragile `grep` shell scripts).
- `list_directory(path)`: Lists the contents of a directory.

#### Agent Execution Loop:

```
[User Prompts Agent]
         │
         ▼
[Call chat.completions.create with Tools]
         │
         ├─► [Response contains content / explanation]
         │        │
         │        ▼
         │   [Stream message to UI] ──► [Finish Turn]
         │
         └─► [Response contains tool_calls]
                  │
                  ▼
             [Render Tool block in UI]
                  │
                  ▼
             [Execute tool (e.g. run command or edit file)]
                  │
                  ▼
             [Feed output back to OpenAI Thread]
                  │
                  └─► (Repeat Loop)
```

### 4. Advanced Agentic Design (Inspiration Improvements)

We will incorporate several critical agentic improvements to optimize performance, context usage, and user experience:

- **Repo Map Context Injection:** Upon starting, the agent will receive a compressed tree representation of the workspace (respecting `.gitignore` exclusions) to instantly understand the repository structure without manual file listing.
- **Dynamic System Prompt Context:** We will dynamically inject current OS details, target shell shell-type, and the absolute current working directory into the agent system prompt on each request.
- **Context Truncation & Token Tracking:** To prevent context window bloat and manage cost/latency, we will implement token tracking and a rolling window/forgetting strategy for older file read tool responses.
- **Separation of Reasoning (<thinking> tags):** The agent will be instructed to wrap its internal thinking process inside `<thinking>...</thinking>` tags. The UI will render these in a collapsible or hidden area to keep the terminal output clean and noise-free.
- **Error Loop Halting:** The agent system will track tool execution failures. If a command fails 3 times with similar errors, the agent loop will stop tool execution and prompt the user for manual intervention.
- **Visual File Edit Diffing:** Instead of a generic `✦ Writing file.js` message, the renderer will parse and display a clean, colorized line diff showing exactly what lines the `edit_file` tool added or deleted. **Note that this diff is treated as standard command/tool output and will follow the active output collapse mode settings (Full, Collapsed, or Last).**

### 5. Multi-State Output Collapse Modes

To manage layout clean-up when outputs are long or verbose, we will support three collapse modes:

- **Full (Normal):** Shows all command output blocks completely (standard behavior).
- **Collapsed:** Hides all command outputs and replaces them with a semi-transparent, readable line-count indicator (e.g., `[42 lines of output]`).
- **Last:** Collapses all historical outputs to their line-count indicators, leaving only the very last `chat-block`'s output expanded.
- **Key Bind Toggle:** Pressing a dedicated shortcut (e.g. `Ctrl+H` / `Cmd+H`) will cycle the layout state: `Full` ➔ `Collapsed` ➔ `Last` ➔ `Full`, dynamically updating rendering classes and layout elements.

### 6. Automatic Input Type & Prompt Detection

To provide immediate visual feedback on how an input line will be processed:

- **Dynamic Parsing:** As the user types, an input event handler checks if the text represents a system command (starts with known binaries like `cd`, `ls`, `git`, or file paths like `./`) or natural language.
- **Visual Indicator Toggle:** If the heuristic detects a natural language query (AI prompt), the input prompt's chevron icon dynamically turns purple (`var(--purple)`). If a shell command is detected, the chevron remains green (`var(--green)`).

### 7. Single-Instance Multi-Terminal Support

To allow spawning multiple independent terminal windows inside a single running Electron instance:

- **App Single Instance Lock:** We use `app.requestSingleInstanceLock()`. If a second process is launched (e.g., running `electron .` again from another terminal), it is blocked, registers the launch request with the primary instance, and immediately exits.
- **Primary Instance listener:** The primary instance listens for the `second-instance` event. When triggered, it receives the CLI arguments and target working directory of the new run, and spawns a new window starting in that directory.
- **Isolated Session Map:** The main process maps IPC interactions to the active window using `event.sender.id`, maintaining completely isolated terminal shells, working directories, and OpenAI chat histories for each window.

### 8. Interactive Slash Commands Autocomplete Box

To provide tab-completion and keyboard navigation for available slash commands directly in the input area:

- **Command Registry:** We define an array of supported command objects `slashCommands = [{ name, description, callback }]` (e.g. `/exit`, `/clear`, `/model`...).
- **Suggestions:** If the user types something that starts with `/`, we will display matching command names and descriptions, highlighting the best match in chat blocks under the input line.
- **Navigation & Autocomplete:**
    - `ArrowUp` / `ArrowDown`: If the suggestion overlay is open, these keys navigate the matches instead of cycling shell history.
    - `Enter`: If the suggestion overlay is open, pressing Enter immediately auto-completes the input field with the highlighted command name and executes its callback.

## Step-by-Step Implementation Roadmap

### Phase 1: Single Instance Lock & Shell Spawning

- Set up `app.requestSingleInstanceLock()` and `second-instance` handlers in `main.js` to enable multi-terminal window spawning.
- Maintain a session mapping registry keying shell processes, buffers, and directories to `event.sender.id`.
- Implement `preload.js` defining IPC channels for command inputs and stdout/stderr pipes.
- Setup the persistent bash shell in `main.js` and implement the line-buffer stdout/stderr parser.
- Configure `window/window.js` to append blocks to `<terminal-chat>` and support arrow-key navigation for command history.
- Integrate the dynamic input listener inside the renderer to automatically toggle the prompt chevron color between green and purple based on command-type heuristics.
- Register `slashCommands`, and wire keyboard navigation interceptors for `/` inputs with suggestion lines at the bottom of the terminal.

### Phase 2: OpenAI SDK Integration & Core Agent Loop

- Install the official `openai` package.
- Setup API key storage. We will support loading `process.env.OPENAI_API_KEY` or saving it locally to `~/.nono-terminal-config.json` via a `/api-key <key>` custom command.
- Implement the agent loop in `main.js` with the tools list (`execute_command`, `read_file`, `edit_file`, `search_codebase`, `list_directory`).
- Integrate the Repo Map generator, dynamic system instructions, and error loop checks.
- Add the `AbortController` timeout wrapper and retry loop around all OpenAI API calls.

### Phase 3: Agentic UI & Visual Enhancements

- Build UI elements for folding reasoning blocks (`<thinking>`) inside `window/window.js`.
- Implement visual styling for the foldout reasoning summaries and contents in `window/style.css`.
- Add renderer support for colorized file edit diff indicators (adding green/red color classes for unified diff additions/deletions).
- Implement the multi-state output collapse selector (`Ctrl+H` key binds, container class toggling, and output placeholder elements).
